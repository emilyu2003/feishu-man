# 虚拟招聘系统 - 竞赛项目说明文档

## 一、项目概述

本项目是一个基于 **LangGraph** 和 **飞书多维表格 (Bitable)** 构建的多智能体协同招聘系统。系统模拟真实招聘流程，通过 AI 驱动的多智能体协作，实现从简历生成、筛选、面试安排到结果评估的全流程自动化。

---

## 二、项目亮点分析

### 1. 已实现的亮点

#### 1.1 多 Agent 协同架构（竞赛要求：≥3个角色）

本系统实现了 **4 个核心 Agent**，完全符合赛事要求：

| Agent | 职责 | 协作关系 |
|-------|------|----------|
| **HR Agent** | 简历筛选、面试安排、Offer发放、报告生成 | 协调者角色，驱动整个流程 |
| **面试官 Agent** | 设置可面试时间、出题、面试评估 | 被 HR Agent 调用 |
| **候选人池 Agent** | 批量生成候选人简历 | 为 HR Agent 提供候选人 |
| **候选人 Agent** | 确认面试、答题、接受/拒绝Offer | 响应面试官和 HR Agent |

#### 1.2 完整业务流程闭环

系统实现了 **8 个流程节点**，形成完整链路：

```
初始化建表 → 简历生成 → 时间设置 → 简历筛选 → 面试安排
    ↓
候选人确认 → 面试执行 → Offer决策 → 数据分析
```

**关键特性**：
- 状态驱动：每个节点执行后更新飞书表格状态
- 数据流转：所有数据通过飞书多维表格传递和共享
- 断点续跑：支持从中断点恢复执行

#### 1.3 基于 LangGraph 的工作流编排（竞赛技术要求）

**技术亮点**：
- 使用 **StateGraph** 定义完整招聘流程
- 支持**异步消息传递**和**状态持久化**
- 节点间通过**共享状态**而非直接调用实现解耦
- 相比 CrewAI/AutoGen，提供更细粒度的控制

**代码实现优势**：
```python
# 每个节点是独立的异步函数，便于调试和扩展
workflow.add_node("initialize", self.node_initialize)
workflow.add_node("screening", self.node_screening)
workflow.add_node("interviewing", self.node_interviewing)

# 边定义清晰，流程一目了然
workflow.add_edge("screening", "scheduling")
workflow.add_edge("scheduling", "confirm_interview")
```

#### 1.4 飞书多维表格深度集成（竞赛核心要求）

**实现的表格结构**：

| 表格 | 用途 | 表间关系 |
|------|------|----------|
| 岗位描述 | 存储 JD 和招聘配置 | 被所有 Agent 读取 |
| 简历池 | 候选人全生命周期管理 | 主数据表 |
| 面试官可用时间 | 时间槽管理 | 与面试安排一对多 |
| 面试邀约记录 | 候选人时间选择 | 状态变更触发器 |
| 面试安排 | 面试执行记录 | 核心业务表 |
| 招聘数据分析 | 报告存储 | HR Agent 输出 |

**技术实现**：
- ✅ 通过 **lark-cli** 原生调用飞书 API
- ✅ 自动检测并创建表格结构
- ✅ 支持增删改查全操作
- ✅ 状态字段驱动流程流转

#### 1.5 智能简历筛选系统

**实现方式**：
- LLM 语义匹配：对比 JD 与简历内容计算相似度评分
- 结构化输出：使用 Pydantic 保证数据格式正确
- 批量处理：支持一次性筛选多名候选人

**评分体系**：
```
相似度评分 = LLM(简历内容, JD要求) → 0-100分
筛选标准 = 评分 ≥ 60分 → 通过/不通过
```

#### 1.6 循环面试时间分配算法

**解决的问题**：
- 候选人数量多于时间槽时的分配问题
- 候选人拒绝后时间槽的回收再利用
- 确保所有候选人都有机会获得面试时间

**算法特点**：
- While 循环持续分配，直到所有候选人都处理或没有更多时间槽
- 拒绝的时间槽自动恢复为"可用"状态
- 最大循环次数保护，防止无限循环

#### 1.7 AI 面试双模式

| 模式 | 适用场景 | API 调用次数 | 面试质量 |
|------|----------|--------------|----------|
| **MCQ 模式** | 快速演示、测试 | 低（预生成题库） | 中等 |
| **交互式模式** | 真实面试 | 高（多轮对话） | 高 |

#### 1.8 批次化招聘与时间逻辑

**实现特性**：
- 支持多批次招聘，自动累计录用人数
- 每批次独立时间线，避免数据混淆
- 批次结束时间 = 最后一场面试时间 + 1小时
- 下一批次起始时间 = 本批次最晚面试日期 + 1天

## 三、核心代码展示

### 3.1 LangGraph 工作流定义 (`src/core/graph.py`)

```python
class RecruitmentWorkflow:
    def __init__(self, feishu: FeishuClient, llm: LLMService):
        self.feishu = feishu
        self.llm = llm
        self.hr = HRAgent(llm)
        self.interviewer = InterviewerAgent(llm)
        self.candidate = CandidateAgent(llm)
        
        # 定义状态图
        workflow = StateGraph(RecruitmentState)
        
        # 添加节点（8个流程节点）
        workflow.add_node("initialize", self.node_initialize)       # 初始化建表
        workflow.add_node("generate_candidates", self.node_generate_candidates)  # 简历生成
        workflow.add_node("set_availability", self.node_set_availability)        # 时间设置
        workflow.add_node("screening", self.node_screening)         # 简历筛选
        workflow.add_node("scheduling", self.node_scheduling)       # 面试安排
        workflow.add_node("confirm_interview", self.node_confirm_interview)      # 候选人确认
        workflow.add_node("interviewing", self.node_interviewing)   # 面试执行
        workflow.add_node("offer_decision", self.node_offer_decision)           # Offer决策
        workflow.add_node("reporting", self.node_reporting)         # 生成报告
        
        # 设置边（流程顺序）
        workflow.set_entry_point("initialize")
        workflow.add_edge("initialize", "generate_candidates")
        workflow.add_edge("generate_candidates", "set_availability")
        workflow.add_edge("set_availability", "screening")
        workflow.add_edge("screening", "scheduling")
        workflow.add_edge("scheduling", "confirm_interview")
        workflow.add_edge("confirm_interview", "interviewing")
        workflow.add_edge("interviewing", "offer_decision")
        workflow.add_edge("offer_decision", "reporting")
        workflow.add_edge("reporting", END)
        
        self.app = workflow.compile()
```

### 3.2 AI 面试执行逻辑 (`src/core/graph.py`)

```python
async def node_interviewing(self, state: RecruitmentState):
    """AI 面试执行节点"""
    jd = state["jd"]
    confirmed_interviews = [inv for inv in state["interviews"]
                           if inv["安排状态"] == "已确认" and inv["面试状态"] == "待进行"]
    
    # 按时间排序，确保面试顺序正确
    confirmed_interviews.sort(key=lambda x: datetime.strptime(x["面试时间"], "%Y-%m-%d %H:%M"))
    
    # MCQ模式：预先生成题库，减少API调用
    all_questions = await self.interviewer.generate_questions(jd, num_questions=10)
    
    for inv in confirmed_interviews:
        questions = all_questions
        answers = await self.candidate.answer_questions(questions)
        result = self.interviewer.evaluate_performance(questions, answers)
        
        inv["面试反馈"] = result["feedback"]
        inv["面试得分"] = result["score"]
        inv["是否通过面试"] = "是" if result["score"] >= 30 else "否"
    
    return {**state, "interviews": confirmed_interviews}
```

class FeishuClient:
    def __init__(self, app_id, app_secret, base_id):
        self.app_id = app_id
        self.app_secret = app_secret
        self.base_id = base_id
        
        # 智能判定 Mock 模式
        self.is_mock = (
            not (app_id and app_secret and base_id) or 
            "xxx" in (app_id or "")
        )
        
        if self.is_mock:
            print("Running in MOCK mode - no Feishu connection")
    
    def _run_cli(self, args):
        """封装 lark-cli 调用，支持本地配置隔离"""
        cmd = ["lark-cli"] + args
        env = os.environ.copy()
        env["USERPROFILE"] = self.local_home
        result = subprocess.run(cmd, capture_output=True, text=True, check=True, env=env)
        return json.loads(result.stdout)
```

### 4.3 核心模型选型思路

```
┌──────────────────────────────────────────────────────┐
│                    模型选型架构                       │
├──────────────────────────────────────────────────────┤
│  LLM Service (LangChain ChatOpenAI)                 │
│       ↓                                              │
│  ┌──────────┐  ┌──────────────┐  ┌──────────────┐   │
